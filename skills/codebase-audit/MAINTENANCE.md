# Contributing to the codebase-audit skill

This file is for **skill maintainers** — people editing the audit playbook
itself, not people running audits with it.

The skill is large (29 modules, ~52 reference files, 14 compliance-framework
sub-references, 12 product-type sub-references). It is also exposed via the
`/audit` slash command, which means changes here are surfaced to every user
of the command. Treat edits with the same discipline you would treat a CLI
release.

---

## Numbering policy

**`modules.json` at the repo root is the authoritative source of truth for
the module catalog** (numbers, names, output paths, reference paths). The
SKILL.md Step 2 catalog table is a *rendered view* of `modules.json`,
regenerated by `tools/render-from-json.ps1` between the
`<!-- BEGIN: modules-catalog -->` / `<!-- END: modules-catalog -->`
markers. The `/audit` command's argument tables and the SKILL.md
description frontmatter are still maintained by hand and cross-validated
by the linter (Invariants 2 + 4).

Workflow when changing the catalog:

1. Edit `modules.json`.
2. Run `tools/render-from-json.ps1` (no flags) to regenerate the SKILL.md
   block.
3. Update `/audit` command-file tables and SKILL.md prose by hand to match.
4. Run `tools/lint-drift.ps1` — all 6 invariants must pass.
5. CHANGELOG entry.

### H1 convention

Every top-level reference file's H1 must include its catalog module number,
formatted as:

```
# <NN>[.<S>] — <Module Name>
```

Examples:

- `# 02 — Security Audit`
- `# 00.3 — Remediation Plan (Ticket Emitter)`
- `# 01.5 — Stack-Specific Brief`

Use two-digit numbers (`02`, not `2`) for sort-stability. Sub-references
(under `compliance-frameworks/` and `product-types/`) do **not** carry module
numbers — they carry the framework or platform name. The dispatcher above
them owns the module number.

### Cross-module reference convention

When one module references another in prose, prefer the form
`Module NN (Name)` so a reader can find the target even if the number
later drifts. For new content, prefer file-path references where they
read naturally — `references/04-performance/speed.md` is rename-stable
even when numbers change. See `ai-ml.md` and `dependencies.md` for
existing examples of the path style.

Bare digits like "module 5" or "module 12" without context are the source
of most prior drift; avoid them.

### Renumbering checklist

When a module's catalog number changes, update — in the same commit —
every place the number appears:

- [ ] `modules.json` (the source of truth)
- [ ] Run `tools/render-from-json.ps1` to regenerate the SKILL.md catalog block
- [ ] `SKILL.md` Step 4 sequencing diagram
- [ ] `SKILL.md` description frontmatter (count + enumeration)
- [ ] The H1 of the affected reference file(s)
- [ ] All cross-references in `references/**/*.md` (grep for the old number)
- [ ] `/audit` command file at `~/.claude/commands/audit.md` (or equivalent on the maintainer's machine): module-name table, named-modes table, examples
- [ ] `CHANGELOG.md` entry describing the change

Then run the linter (`tools/lint-drift.ps1`) to verify nothing was missed
across SKILL.md, references/, CONTRIBUTING.md, and the `/audit` command
file. The linter validates: (1) path resolution; (2) H1 ↔ catalog parity;
(3) Live Discovery bidirectional consistency; (4) audit.md ↔ SKILL.md
sync; (5) output-template fields (commit hash + Final Recommendation +
Decision Summary present in every finding-producing module); (6)
modules.json ↔ SKILL.md catalog parity.

### Adding a new module

- [ ] Pick the next available catalog number in the relevant category
- [ ] Add the row to `SKILL.md` Step 2 catalog table
- [ ] Add to `SKILL.md` Step 4 sequencing if it has a fixed slot
- [ ] Add to `SKILL.md` description enumeration; bump the dimension count
- [ ] Create `references/<category>/<file>.md` with the H1 in the
      `# NN — Name` convention
- [ ] Add to `/audit` command file: module-name table + appropriate examples
- [ ] If it has Live Discovery, add to the SKILL.md Step 1.5 module list
- [ ] If finding-producing, add to the Module 00.3 walk list and 00.2
      taxonomy
- [ ] CHANGELOG entry

### Deprecating a module

- [ ] Mark deprecated in the catalog row with the date and replacement (if any)
- [ ] Update cross-references
- [ ] Do not silently delete — leave the reference file with a deprecation
      header so users following old links land somewhere

## Anti-rot: Live Discovery modules

Modules whose findings depend on fast-moving facts have a `Step 0: Live
Discovery` section. The current list lives in `SKILL.md` Step 1.5.

When adding Live Discovery to a module:

- Cite at least one authoritative source URL per discovered fact
- Specify what to capture (versions, dates, thresholds, model IDs)
- Specify the fallback when network is unavailable
- Add the module to the SKILL.md Step 1.5 list

When the underlying source moves (URL change, page renamed, source
deprecated): update the module's Step 0 references in the same release.

## Output discipline

Every finding-producing module should provide, at minimum:

- **Live Discovery** (Step 0) where the dimension rots
- **Investigation steps** with concrete grep patterns, tooling pointers,
  and file paths
- **Red flags** — short list of high-signal anti-patterns
- **Output structure** — explicit markdown template with named sections
- **Severity calibration** — 4-tier (Critical / High / Medium / Low) with
  examples
- **Evidence requirements** — every Critical / High finding cites a
  file:line; absence-based findings name what was searched and where
- **Final Recommendation** + **Decision Summary** blocks where applicable

Synthesis modules (00.1–00.4) and dispatcher modules (01.5, 19, 22) deviate
from this shape intentionally — they aggregate or dispatch rather than
produce findings directly.

Product-type and compliance-framework **sub-references** inherit severity
calibration and evidence requirements from their dispatcher (see the
"Sub-reference inheritance" note in `product-type.md`). Do not duplicate
those sections in sub-refs.

## CHANGELOG convention

Date-based versioning. Each meaningful change to the playbook gets an
entry. Minor wording fixes don't.

Entry structure:

```markdown
## YYYY-MM-DD (optional descriptor)

Brief context paragraph if the release ties together multiple changes.

### Added
- ...

### Fixed
- ...

### Reorganized
- ...

### Known gaps (next maintenance pass)
- ...
```

Update `VERSION` in the same commit to match the entry date.

## Style

The "A note on style" section at the bottom of `SKILL.md` applies to skill
maintainers too:

- Honest, actionable; no checklist theater
- A short sharp module with 5 well-evidenced patterns beats 30 pages of
  generic platitudes
- A boring "keep it" recommendation is often correct
- Calibrate to the project being audited, not to an imagined enterprise
  org chart

When adding content, ask: would a maintainer of the projects this skill
audits actually use this, or am I just filling space? If the latter, cut.
