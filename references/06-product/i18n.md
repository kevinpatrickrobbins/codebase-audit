# 20 — Internationalization (i18n) & Localization (l10n)

Audit the project's ability to serve users in their language and locale —
or its readiness to add that capability.

Output: `/docs/audits/i18n-audit.md`

---

## When to apply

Run when the project either:

- Currently targets non-English markets
- Plans to expand to non-English markets
- Has explicit English-only positioning that should be documented as a
  conscious choice

For a US-only B2B internal tool serving English speakers exclusively,
findings here are advisory rather than required. Skip the depth.

- **Frontend Modernization** (Module 17) covers logical CSS properties

## Step 0: Live Discovery

Locale standards and tooling evolve.

- **CLDR (Unicode Common Locale Data Repository):** Fetch
  https://cldr.unicode.org/index/downloads — current locale support and
  pluralization rules.
- **Intl API:** Fetch
  https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Intl —
  current Intl.* support across runtimes.
- **ICU MessageFormat 2.0:** Fetch
  https://github.com/unicode-org/message-format-wg — successor to ICU
  MessageFormat; check current state.
- **Framework i18n libraries:** `Bash` `npm view next-intl version`,
  `npm view react-intl version`, `npm view i18next version`,
  `npm view vue-i18n version`, `npm view svelte-i18n version`,
  `npm view @nuxtjs/i18n version`.

## Step 1: Language reach detection

- **Stated targets.** README, marketing copy mention non-English markets?
- **Locale config.** `next.config.js` `i18n.locales`, `nuxt.config`
  `i18n`, `astro.config` `i18n`, similar.
- **Translation files.** `locales/`, `messages/`, `i18n/` directory with
  per-locale files.
- **CLDR-style locale codes** — `en-US`, `pt-BR`, `zh-Hant` properly used
  vs lazy `en`, `pt`, `zh` (the latter loses information about the
  variant).

## Step 2: String externalization

- **Hardcoded English strings** in components, server code, error
  messages — anything user-visible must be externalized.
- **Concatenated translations** — `t('hello') + ' ' + name + '!'` —
  breaks in languages where word order differs. Use placeholder
  interpolation: `t('hello_name', { name })`.
- **Coverage estimate.** What percentage of user-visible strings are
  externalized vs hardcoded? Spot-sample and extrapolate.

## Step 3: Pluralization

- **ICU MessageFormat or library equivalent** for plural rules.
- **Slavic-language plurals** (Russian, Polish, Ukrainian) require 3
  forms; "n=1 ? singular : plural" English logic fails for these.
- **Library check:** does the chosen i18n lib support full CLDR plural
  rules?
- **Gendered translations** — some languages need different forms by
  subject gender.

## Step 4: Date / number / currency formatting

- **`Intl.DateTimeFormat`** for dates — `Intl.NumberFormat` for numbers
  / currency. Avoid hardcoded format strings.
- **Currency rendering** — symbol position varies (`$10` vs `10 €`);
  decimal separator varies.
- **Number formatting** — `1,234.56` (en-US) vs `1.234,56` (de-DE) vs
  `1 234,56` (fr-FR with NBSP).
- **Calendar systems** — Hijri / Buddhist / Japanese era calendars in
  some markets.
- **Relative time** — `Intl.RelativeTimeFormat` for "3 days ago".
- **List formatting** — `Intl.ListFormat` for "A, B, and C" / "A, B y C".

## Step 5: RTL (right-to-left)

- **Logical CSS properties** (cross-ref Module 17) — `margin-inline-start`
  vs `margin-left`. RTL languages (Arabic, Hebrew, Persian, Urdu) need
  the layout to flip.
- **`dir="rtl"`** attribute applied at root for RTL locales.
- **Icons / directional UI.** Arrows, breadcrumbs, sliders — flip in RTL?
  Search for chevron / arrow icons that wouldn't auto-flip.
- **Text alignment** — `text-align: start` not `left`.
- **Bidirectional text handling** — RTL users may have LTR phone numbers,
  email addresses inside Arabic prose. `<bdi>`, `unicode-bidi`,
  `direction` correctly used.

## Step 6: Locale detection & routing

- **Detection sources.** Accept-Language header, user setting,
  geolocation IP, subdomain. Override hierarchy documented?
- **URL strategy.** Subdomain (`fr.example.com`), path
  (`example.com/fr`), query (`?lang=fr`), domain (`example.fr`). Each has
  SEO trade-offs (cross-ref SEO module).
- **Default locale fallback.** When user's locale isn't supported, what
  shows? English by default? Closest available?
- **`Accept-Language` parsed correctly** — `q=` weights, multi-locale
  fallback chain.
- **Locale switcher UI** — visible, accessible, persists choice.

## Step 7: Translation workflow

- **Source of truth.** Code as source (`t('key')` calls extracted) or
  translation files as source (designers/PMs edit JSON)?
- **Tools:** Crowdin, Lokalise, Phrase, Weblate (open source), Transifex,
  Tolgee.
- **Pseudo-localization** in dev/test — `[¡Hølló Wörld!]` with brackets
  — catches non-externalized strings and layout overflow issues.
- **Translation lag.** When new strings ship, how long until translations
  arrive? Default-to-source-language during gap?

## Step 8: Quality gates

- **Translation completeness** in CI — fail if a non-default-locale file
  has missing keys?
- **No hardcoded English** lint rule (`eslint-plugin-i18next`,
  `eslint-plugin-i18n-text`, or similar)?
- **RTL visual regression** in test suite (Playwright with `dir=rtl`)?
- **Locale-specific test coverage** — auth flows, payment flows tested in
  the languages claimed?

---

## Red flags

- README claims international support but every string is hardcoded
  English.
- `t('hello') + ' ' + name + '!'` patterns (concatenated translation).
- `new Date().toLocaleDateString()` without locale arg.
- `dir="ltr"` hardcoded in HTML.
- Translation files only contain English (the project never set up any
  other locale despite claims).
- Currency rendered as `$` regardless of locale.
- No RTL support despite Arabic / Hebrew users.
- Subdomain-based locale routing without `hreflang` in HTML.
- Locale codes lazy: `en`, `pt`, `zh` rather than `en-US`, `pt-BR`,
  `zh-Hant`.
- "Welcome, " + name + "!" (gendered forms ignored).

## Severity calibration

- **Critical:** product claims international support but core flows
  (signup, payment) fail in target locales.
- **High:** hardcoded English in user-visible strings; missing
  pluralization for languages with > 2 forms; no RTL support for stated
  RTL markets.
- **Medium:** sub-optimal locale codes; no pseudo-localization in test;
  no lint rule against hardcoded text.
- **Low:** style consistency; minor translation gaps.

Every finding cites the file/line and the affected locale(s).
